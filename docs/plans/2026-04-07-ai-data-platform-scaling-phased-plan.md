# AI Data Platform 分阶段提效与扩容方案

## 目标

为 `ai-data-platform` 制定一份从当前单项目可用状态，逐步演进到稳定支持更高并发的正式方案。

本方案分为 3 个阶段：

1. 优化提效与稳定
2. 扩容到 100 人在线
3. 扩容到 1000 人在线

这里的“在线”定义为：

1. 有活跃页面访问、问答请求、文档查看、报表操作、机器人入口消息的并发在线用户
2. 不等同于“1000 人同时跑 OCR / 重报表 / 重向量重建”

如果目标是大批量重任务同时发生，需要在第 3 阶段上进一步增加专门的异步算力和队列隔离。

---

## 当前基线

当前项目已经具备较强的单项目产品能力，但在部署与运行模型上仍然偏向“单机强化版应用”：

1. 主链已经形成：
   - 文档接入
   - quick / detailed 分层解析
   - 结构化字段、证据块、表格洞察
   - memory-first 问答
   - 报表中心
   - 企业微信机器人入口

2. 当前优势：
   - 知识资产层较强，问答与报表共用一套底座
   - 文档解析治理已经具备知识库级模板、别名、反馈回流
   - 文本规范库新增了库级编译摘要层

3. 当前主要限制：
   - 仍有本地文件状态和运行态配置
   - 解析、报表、向量、dataviz 等长任务与在线请求仍有竞争关系
   - Web / API / Worker / Model Bridge 还没有完整的横向扩容边界
   - 静态资源、缓存、队列、观测体系还不够生产化

---

## 总体判断

建议不要直接跳到“1000 人在线架构改造”。

更合理的方式是：

1. 先做一轮提效和稳定性收口
2. 再把系统提升到 100 人在线的稳定可运营状态
3. 最后再做 1000 人在线的基础设施和服务拆层

原因很简单：

1. 当前系统的最大价值在知识资产层，不在入口数量
2. 现阶段最该保护的是主链稳定性，而不是过早微服务化
3. 直接按 1000 人在线目标改，容易做出过重架构，反而拖慢主线

---

## 架构红线

扩容过程中要守住下面几条红线：

1. 不重做知识资产主链  
   现有 `quick parse -> detailed parse -> memory/vector sync -> compiled summary` 主链是对的，后续只做拆层和外置化，不推翻。

2. 不让在线请求直接承担长任务  
   OCR、重解析、向量重建、报表图表渲染、库级编译等都应逐步彻底异步化。

3. 不把 Markdown / 本地 JSON 当生产级共享状态  
   编译结果、运行态配置、反馈状态、审计统计要逐步迁到可共享的正式存储。

4. 不把“扩容”理解成只加机器  
   真正的瓶颈往往是：
   - 状态共享
   - 队列隔离
   - 缓存命中
   - 流量治理
   - 可观测性

## 阶段一：优化提效与稳定

第一阶段的可执行拆解见：

### 目标

在不改变整体产品边界的前提下，先把当前系统的关键主链做稳、做轻、做可观测。

### 目标结果

1. 页面加载、问答响应、文档中心、报表中心明显更稳
2. 长任务不会轻易拖垮在线请求
3. 线上问题更容易定位
4. 为后续 100 人在线扩容建立清晰边界

### 核心改造

#### 1. 静态资源与 Web 交付优化

1. 给 `/_next/static/`、图片、导出资产单独做稳定反代和缓存策略
2. 避免普通页面与 websocket/upgrade 头混用
3. 将大页面拆成更稳定的前端资源加载模式，减少 chunk 超时

#### 2. API 主链提效

1. 收紧首页、文档中心、报表中心的首屏接口数量
2. 为高频读取接口增加轻量缓存
3. 把慢查询、重复读取、重复编译结果做本地缓存或共享缓存预埋

#### 3. 长任务异步化收口

优先隔离：

1. 文档详细解析
2. 向量重建
3. 库级编译摘要更新
4. 报表图表渲染

第一阶段可以先不引入复杂消息系统，但至少要保证：

1. 在线请求只负责任务投递
2. Worker 独立消费
3. 有任务状态和失败重试

#### 4. 运行态状态收口

1. 把本地运行态 JSON 继续拆成“默认配置 + 本地覆盖 + 可共享状态”
2. 明确哪些文件只是开发运行态，哪些应进入正式共享存储
3. 尽量减少“单机本地文件就是生产状态”的情况

#### 5. 观测与审计增强

至少补齐：

1. 请求耗时
2. 队列长度
3. 解析耗时
4. 报表生成耗时
5. 图表渲染耗时
6. 失败率
7. 外部渠道消息成功率

### 交付周期

建议周期：**2 到 3 周**

### 阶段完成标志

1. 文档中心、报表中心、首页访问稳定
2. 长任务不会显著拖慢在线请求
3. 常见故障能通过日志和指标快速定位
4. 后续扩容改造不再依赖猜测

## 阶段二：扩容到 100 人在线

### 目标

让系统在真实业务场景下稳定支撑一个团队级、部门级或小范围客户级使用。

这个阶段的重点不是“极限高并发”，而是：

1. 多人同时问答
2. 多人同时浏览文档和报表
3. 有一定数量机器人消息并发
4. 后台持续存在解析/报表任务

### 目标结果

1. 100 人在线情况下前台可用
2. 重任务对前台影响可控
3. 单机故障不至于整站不可用

### 核心改造

#### 1. Web / API / Worker 分实例

从“单机一体化”升级为：

1. Web 独立实例
2. API 独立实例
3. Worker 独立实例
4. Model Bridge 独立实例

这一步不要求微服务化，但要求进程边界清楚。

#### 2. Redis 引入

Redis 在这个阶段非常关键，建议承担：

1. 高频读取缓存
2. 会话态缓存
3. 短期上下文
4. 限流计数
5. 队列和任务状态

#### 3. 共享文件与对象存储

需要把以下内容逐步脱离本地磁盘：

1. 上传文件
2. 报表导出产物
3. 图表渲染产物
4. 可能需要被多实例读取的编译结果

建议至少把上传和输出产物放到共享对象存储。

#### 4. 数据库与状态共享

要明确以下状态进入正式数据库或共享存储：

1. 文档元数据
2. 解析状态
3. 任务状态
4. 审计日志
5. 满意度反馈
6. bot 配置与权限

#### 5. 流量治理

这个阶段就要补：

1. 登录/接口限流
2. 重任务队列上限
3. 外部机器人入口节流
4. API 请求超时和重试边界

### 建议部署形态

1. 一个入口层：Nginx 或云负载均衡
2. 2 个 Web 实例
3. 2 个 API 实例
4. 2 组 Worker
   - 文档处理
   - 报表/图表处理
5. 1 套共享 Postgres
6. 1 套 Redis
7. 1 套对象存储

### 交付周期

建议周期：**2 到 4 周**

### 阶段完成标志

1. 100 人在线访问稳定
2. Web / API / Worker 可独立重启
3. 文件和关键状态不依赖单机磁盘
4. 常见峰值不会造成全站明显抖动

## 阶段三：扩容到 1000 人在线

### 目标

把系统从“团队级在线使用”提升到“平台级在线使用”。

这个阶段的重点是：

1. 横向扩展
2. 多实例一致性
3. 高峰流量隔离
4. 故障恢复
5. 大规模观察和运营

### 现实假设

这里的 1000 人在线默认是：

1. 有活跃前台访问
2. 有持续问答请求
3. 有一定比例的数据查看、报表查看、机器人互动
4. 重解析和重报表不是所有人同时触发

如果要求是：

1. 1000 人同时发起复杂问答
2. 大量用户同时跑 OCR、重解析、重报表

那实际上已经接近“高算力 AI 平台”问题，资源和队列都需要再上一个量级。

### 核心改造

#### 1. 服务拓扑正式化

建议目标拓扑：

1. CDN
2. 负载均衡
3. Web 多实例
4. API 多实例
5. Worker 多池
   - 解析池
   - 向量池
   - 报表池
   - dataviz 池
   - 渠道机器人池
6. 模型网关/桥接独立扩容

#### 2. 队列系统升级

此时建议从轻量任务队列升级为真正可观测的消息队列或更稳的任务系统，至少满足：

1. 多队列
2. 优先级
3. 重试
4. 死信
5. 延迟任务
6. 幂等消费

#### 3. 向量与检索能力升级

1000 人在线下，规则召回 + 向量召回的压力会明显上升。

要考虑：

1. 向量索引服务独立化
2. 热点知识库缓存
3. 召回结果缓存
4. 别名加权和 focus 字段打分优化

#### 4. 更严格的资源隔离

要把以下资源分层：

1. 在线问答计算资源
2. 文档解析计算资源
3. 报表生成计算资源
4. 图表渲染计算资源

避免高峰期被单一任务类型拖垮。

#### 5. 观测和发布体系升级

必须具备：

1. 指标监控
2. 分布式 tracing
3. 告警
4. 灰度发布
5. 回滚
6. 故障演练
7. 容量压测报告

### 建议部署形态

1. 入口层：
   - CDN
   - 负载均衡

2. 应用层：
   - 2 到 4 个 Web 实例
   - 3 到 6 个 API 实例

3. 任务层：
   - 多 Worker 池
   - 按任务类型拆分

4. 状态层：
   - 高可用 Postgres
   - Redis
   - 对象存储
   - 向量索引服务

### 交付周期

建议周期：**6 到 8 周**

如果需要同时完成：

1. 基础设施升级
2. 代码链路改造
3. 压测
4. 故障演练
5. 生产切换

那么实际节奏更接近 **8 到 10 周** 会更稳。

### 阶段完成标志

1. 1000 人在线访问稳定
2. 单实例故障不会导致整站不可用
3. 队列积压、热点接口、模型网关压力可观测
4. 高峰期具备明确的限流与降级策略

---

## 建议执行顺序

### 先做什么

优先做 **阶段一**。

原因：

1. 当前系统的主要风险不在“机器不够”，而在“状态、队列、长任务与在线请求没有彻底解耦”
2. 如果第一阶段不做，直接冲 100 或 1000，后面只会把问题放大

### 最合理的路线

1. 阶段一：提效和稳定
2. 阶段二：团队级并发能力，稳定到 100 在线
3. 阶段三：平台级并发能力，扩到 1000 在线

---

## 对当前项目的总体建议

这套系统当前最强的是“知识资产层”，不是“入口层”。

所以扩容时要继续坚持：

1. 保护知识资产主链
2. 保护 memory-first 调用链
3. 让报表、问答、机器人继续共用同一底座

不要为了扩容而把系统拆成互相割裂的几套子系统。

---

## 一句话结论

这套系统完全可以往更大规模扩，但正确顺序不是“立刻做 1000 人架构”，而是：

**先稳住主链，再扩到 100 在线，最后再把基础设施和任务体系升级到 1000 在线。**
