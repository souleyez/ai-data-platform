# OpenClaw 双模式能力边界设计

日期：2026-04-01  
状态：设计中  
范围：`ai-data-platform` 本地桌面安装态 / 本机文件处理 / OpenClaw 能力边界

## 背景

当前系统正在朝着 `memory-first + supply-first` 收敛：

- OpenClaw 长期记住文档库、文档卡片、模板库和近期变化。
- 项目侧逐步从“重编排回答”退到“供料、索引、分类、审计、normalize”。
- 用户希望进一步重新定义 OpenClaw 的本机能力边界，不再一刀切。

新的目标不是简单恢复旧版访问密钥门禁，而是把系统能力拆成两种模式：

1. 服务智能模式：默认模式。允许 OpenClaw 搜索、读取、索引、分类、回答、输出，但不允许修改或删除本机系统文件。
2. 完全智能模式：需要输入数字密钥启用。第一次使用时由首个用户设置密钥。启用后恢复 OpenClaw 完整本机能力边界。

这里的关键不是“首页能不能进”，而是“OpenClaw 当前能对本机文件做什么”。

## 目标

### 产品目标

- 默认对所有用户开放服务智能模式，不再用密钥拦整站入口。
- 只有当用户要启用更高本机权限时，才需要输入密钥。
- 让用户明确知道当前处于哪种智能模式。
- 让 OpenClaw 在不同模式下遵守不同能力边界。

### 技术目标

- 让“模式”成为运行时能力边界，而不是纯 UI 状态。
- 让项目侧只做能力约束和供料，不重新接管回答逻辑。
- 与现有 `memory-first` 架构兼容，不回退到旧的全站 gate 模式。

## 推荐方案

推荐采用 **默认服务智能 + 显式升级完全智能** 的双模式架构。

### 方案 A：全站默认服务智能，按需升级完全智能

这是推荐方案。

- 站点默认可直接访问。
- 默认模式下，OpenClaw 只能做“读、搜、复制入库、索引、分类、输出”。
- 当用户显式启用“完全智能模式”时：
  - 如果系统还没有密钥，首个用户创建 4-8 位数字密钥。
  - 如果系统已有密钥，则输入密钥后进入完全智能模式。
- 完全智能模式是“能力升级”，不是“页面解锁”。

优点：

- 和“供料更强、回答更薄”方向一致。
- 默认体验不被门禁打断。
- 高风险能力边界清晰。
- 可以继续保留“分享页免密、对话默认可用”的轻量体验。

缺点：

- 需要单独维护一个模式状态，而不是只有“已登录/未登录”。

### 方案 B：每次高风险动作都单独要密钥

不推荐作为主方案。

- 每次遇到高风险本机动作都弹密钥确认。

缺点：

- 对话会很碎。
- 用户对当前模式没有整体认知。
- 后续很难扩展成清晰的能力边界。

## 双模式能力矩阵

### 服务智能模式

默认允许：

- 搜索本机目录和文件
- 读取本机文件内容
- 复制文件到系统托管区
- 自动索引、结构化、向量化
- 自动分组，命不中则进 `未分组`
- 同步 OpenClaw memory
- 基于文档库回答
- 选择是否按模板输出静态页 / 表格 / 报告
- 下载系统托管副本
- 删除系统托管副本、报告、模板引用等“系统自有资产”

默认禁止：

- 删除本机原文件
- 覆盖本机原文件
- 移动本机原文件
- 重命名本机原文件
- 对本机目录做批量改写

解释：

服务智能模式下，OpenClaw 对本机文件系统的默认边界是 **只读 + 复制入库**。  
也就是“可以看、可以搜、可以收进系统”，但“不碰原件”。

### 完全智能模式

启用条件：

- 用户显式要求启用完全智能模式
- 输入正确数字密钥
- 首次启用时由首个用户创建密钥

允许：

- 服务智能模式下的全部能力
- 恢复 OpenClaw 对本机系统文件的完整处理能力边界

仍然保留的底线：

- 审计排除规则仍然生效
- 项目侧仍然保留结果 normalize
- 高风险破坏性动作仍需用户明确表达意图

解释：

完全智能模式不是“自动允许胡乱删改文件”，而是“不再由系统产品层强行屏蔽 OpenClaw 的完整本机能力”。  
但显式破坏性动作仍需要清晰用户意图，不应 silent destructive。

## 能力边界定义

建议把操作按三层划分：

### 第一层：只读发现

- 搜索本机文件
- 读取目录
- 读取文件
- 摘要、分类、比对、建索引

这层在两种模式都允许。

### 第二层：复制入库

- 复制原文件到系统托管区
- 做结构化、向量化、自动分组
- 更新文档库和 memory

这层在两种模式都允许。

### 第三层：原件改写

- 删除原文件
- 覆盖原文件
- 重命名原文件
- 移动原文件
- 批量整理原文件目录

这层只在完全智能模式开放。

## 与当前架构的关系

这次边界重定义不会推翻当前 `memory-first` 设计，而是叠加在其上。

### 保持不变

- 文档库目录、文档卡片、模板库继续同步进 OpenClaw memory
- 回答继续朝“少编排、多供料”方向收
- 模板仍然由 OpenClaw 自行判断是否采用
- 项目侧继续负责：
  - ingest
  - index
  - classify
  - audit
  - memory sync
  - output normalize

### 新增变化

- 运行时增加 `intelligenceMode = service | full`
- 本机文件相关入口在执行前统一检查当前模式
- 模式信息作为能力边界透传给 OpenClaw

## UI/交互设计

### 首页

首页不再做全站密钥门禁。改成：

- 顶部或侧边显示当前模式：
  - `服务智能模式`
  - `完全智能模式`
- 提供一个“启用完全智能模式”入口

### 首次启用完全智能模式

如果还没有密钥：

- 弹出最小初始化流程
- 用户输入 4-8 位数字密钥
- 保存到本地系统配置

### 再次启用完全智能模式

- 输入已有密钥
- 成功后切换到 `full`

### 降回服务智能模式

- 随时可降回
- 不需要再次输入密钥

## 运行时设计

### 模式状态

建议引入统一状态：

```json
{
  "intelligenceMode": "service | full",
  "fullModeEnabledAt": "optional timestamp",
  "fullModeEnabledBy": "optional local marker"
}
```

项目侧需要感知：

- 当前请求处于哪个模式
- 是否允许执行本机原件改写

### 密钥状态

建议复用之前的极简数字密钥设计，但只用于“完全智能模式升级”，不再负责整站 gate。

建议保留：

- `/api/access-keys/status`
- `/api/access-keys/verify`
- `/api/access-keys`

但语义改成：

- 是否已配置完全智能模式密钥
- 校验是否允许从 `service` 升级到 `full`

### 本机文件能力约束

建议引入统一能力检查：

- `canReadLocalFiles`
- `canImportLocalFiles`
- `canModifyLocalSystemFiles`

映射关系：

- `service`
  - true
  - true
  - false
- `full`
  - true
  - true
  - true

## API/执行层变化建议

### 不需要限制的接口

这些在服务智能模式下继续可用：

- `/api/documents/candidate-sources`
- `/api/documents/candidate-sources/import`
- `/api/documents/scan`
- `/api/documents/local-files/import`
- 文档预览、下载、问答、输出

### 需要新增模式检查的能力

未来所有涉及本机原件改写的能力，统一加模式检查。

如果后续增加：

- `rename local file`
- `move local file`
- `delete local file`

则必须先检查：

- 当前是否为 `full`

### 对 OpenClaw 的传递

项目侧不再重编排回答，但需要给 OpenClaw 一个明确能力边界说明：

- 当前模式是 `service` 还是 `full`
- 当前模式下：
  - 是否只能读 + 复制入库
  - 是否允许改写本机原件

这样 OpenClaw 在对话里就不会误以为自己能直接清理或改写本机文件。

## 安全约束

### 服务智能模式的安全原则

- 默认只读本机原件
- 默认复制入库
- 默认不删改本机系统文件

### 完全智能模式的安全原则

- 恢复完整能力边界
- 但破坏性动作仍要有显式用户意图

### 审计不变

无论哪种模式：

- 审计排除仍然是硬规则
- 不支持的文件格式仍然不入库
- 只解析可解析文件格式

## 实施顺序

### Phase 1：模式与密钥基础设施

- 恢复 access key 存储与校验
- 语义改成“完全智能模式升级密钥”
- 增加 `intelligenceMode` 状态

实现注记（2026-04-01）：

- 已新增 `/api/access-keys/*` 与 `/api/intelligence-mode*` 基础接口。
- 根接口 `/` 与健康检查 `/api/health` 已透出当前 `intelligenceMode` 与能力矩阵。
- 当前阶段仍以运行时状态与能力描述为主，前端切换入口和具体执行层约束将在后续阶段继续接入。

### Phase 2：UI 模式切换

- 首页展示当前模式
- 提供“启用完全智能模式”
- 首次使用可设置密钥

### Phase 3：执行层能力约束

- 增加统一 capability check
- 服务模式默认禁止本机原件改写
- 读、搜、复制入库继续放行

### Phase 4：OpenClaw 模式透传

- 在 prompt / runtime context 中明确当前能力边界
- 不再靠项目侧代替 OpenClaw 做行为假设

## 推荐结论

推荐正式采纳：

- 默认 `服务智能模式`
- 输入数字密钥后升级到 `完全智能模式`
- 恢复完整能力边界的对象是 **OpenClaw 的本机处理权限**，不是整站访问权限
- 继续坚持“项目侧做供料和约束，OpenClaw 做判断和回答”

这条方案和当前 `memory-first + 少编排` 方向一致，也比旧版“整站访问密钥门禁”更合理。
