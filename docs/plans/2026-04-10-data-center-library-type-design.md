# 数据中心与库类型升级设计

Date: 2026-04-10

## 目标

把当前“文档中心”升级为“数据中心”，并给知识库新增一条稳定的顶层维度：

- `libraryType = document`
- `libraryType = data`

这次升级不是单纯改文案，而是把“库里的内容该怎么被解析和供给模型”显式配置出来。

当前系统已经有：

- `bizCategory`
- `schemaType`
- `knowledgePages`
- 文档详细解析
- memory-first 候选选择
- 最新详细解析文档全文供料

但还缺少一个清晰的配置维度去表达：

- 这是一个全文型文档库
- 还是一个结构型数据资料库

## 核心结论

推荐方案：

- 页面从“文档中心”升级为“数据中心”
- “文件库类型”统一改为“库类型”
- 在知识库定义层新增 `libraryType`
- 不新增第二个独立开关
- 直接把库类型映射为默认供料策略

推荐默认映射：

- `document` -> 全文优先
- `data` -> 结构摘要优先，但允许全文供料

这个方案的关键点是：

- 系统只决定“材料如何摆给 OpenClaw”
- 不决定“OpenClaw 必须怎么使用这些材料”
- 模型仍然自己判断如何回答、如何生成报表、如何组织输出

## 为什么不能只改名称

当前“文档中心”虽然名字偏文档，但实际已经承载了多类资料：

- PDF / DOCX / PPT / 网页正文
- CSV / XLSX / JSON
- 报表样本、库存快照、订单样本

如果只把页面名改成“数据中心”，但运行时仍然一律按全文或转 Markdown 供料，会继续出现两个问题：

1. 对文档型资料是对的，但对数据型资料不稳定  
2. 宿主会把结构化数据误处理成一大段文本上下文，降低模型对字段、样本和统计结构的利用率

所以这次升级必须把“命名”和“供料策略”一起收口。

## 术语定义

### 库类型

新增顶层字段：

```ts
type LibraryType = 'document' | 'data';
```

含义：

- `document`
  - 文档型
  - 适合转 Markdown、抽取全文、按正文理解
  - 默认全文优先

- `data`
  - 数据型
  - 适合先展示结构摘要、字段画像、样本行、统计概览、异常点
  - 默认结构摘要优先，但不禁止全文供料

### 和现有字段的关系

- `bizCategory`
  - 回答“业务上属于哪类内容”
  - 例如：`contract`、`order`、`inventory`

- `schemaType`
  - 回答“解析后更像哪种内容结构”
  - 例如：`resume`、`contract`、`report`

- `libraryType`
  - 回答“这个库默认应该怎么供料”

三者不应互相替代。

## 推荐的运行时规则

### 文档型库

默认路径：

- 保持当前详细解析主链
- 保持转 Markdown / 正文抽取 / 全文块供料
- 普通聊天时，全文块优先于结构摘要块

适合内容：

- 招标文件
- 合同协议
- 简历
- 方案文档
- PPT / PDF / 网页正文

### 数据型库

默认路径：

- 仍然允许解析正文和保留全文
- 但聊天和输出时，先供给结构摘要块
- 全文块作为补充上下文，而不是第一顺位

结构摘要块建议包含：

- 字段列表
- 主键或主维度
- 样本行
- 统计概览
- 高频维度
- 异常点
- 最近文件

适合内容：

- 订单明细
- 库存快照
- 指标导出
- 表格型业务样本
- JSON / CSV / XLSX 数据资料

## 推荐的供料优先级

### `document` 库

普通聊天默认供料顺序：

1. 最新详细解析文档全文块
2. memory-first 候选文档上下文
3. 常规检索证据块

### `data` 库

普通聊天默认供料顺序：

1. 结构摘要块
2. memory-first 候选文档上下文
3. 全文块或原始内容块
4. 常规检索证据块

这里的重点不是把全文禁掉，而是把“结构画像”放到前面。

OpenClaw 最终怎么用这些内容，仍由模型自己判断。

## 推荐实现方式

### 方案 A：只改 UI 名称

内容：

- 文档中心 -> 数据中心
- 文件库类型 -> 库类型

优点：

- 最快

缺点：

- 运行时没有变化
- 对数据型资料没有实际改善

结论：

- 不推荐

### 方案 B：新增 `libraryType`，并驱动默认供料策略

内容：

- 库新增 `libraryType`
- 页面展示和编辑该字段
- 运行时按库类型决定供料优先级

优点：

- 改动可控
- 不拆现有产品结构
- 最贴合当前代码和用户预期

缺点：

- 需要调整库定义、文档中心设置、聊天供料和输出供料

结论：

- 推荐

### 方案 C：文档中心 / 数据中心拆成两套独立系统

内容：

- 页面、库定义、上传链、解析链、供料链全部分叉

优点：

- 概念最彻底

缺点：

- 成本高
- 容易把现有知识库和输出链打碎

结论：

- 当前阶段不推荐

## 代码落点

### 1. 知识库定义层

文件：

- `apps/api/src/lib/document-libraries.ts`
- `apps/api/src/routes/documents.ts`

变更：

- `DocumentLibrary` 新增 `libraryType?: 'document' | 'data'`
- 创建库、更新库接口允许读写该字段
- 默认值设为 `document`

### 2. 数据中心页面

文件：

- `apps/web/app/documents/page.js`
- `apps/web/app/documents/api.js`

变更：

- 页面标题从“文档中心”升级为“数据中心”
- “知识库分组”相关文案统一升级为“库”或“数据库”
- 编辑区新增“库类型”
- 新建库时可指定类型，默认 `document`

### 3. 聊天供料层

文件：

- `apps/api/src/lib/knowledge-chat-dispatch.ts`
- `apps/api/src/lib/knowledge-supply.ts`

变更：

- `document` 库继续沿用全文优先
- `data` 库新增结构摘要优先块
- 最新详细解析文档全文供料改为“按当前命中的库类型决定顺序”

### 4. 报表输出层

文件：

- `apps/api/src/lib/report-center.ts`
- `apps/api/src/lib/knowledge-output.ts`

变更：

- 输出时也复用 `libraryType` 供料优先级
- 数据型库优先提供结构摘要与字段证据
- 文档型库继续优先按正文组织内容

## 推荐的结构摘要内容

对于 `data` 库，建议统一生成一个轻量结构摘要块，形态如下：

- 文件名
- 文件类型
- 数据主维度
- 字段列表
- 行数 / 列数 / sheet 数
- 样本行
- 核心统计摘要
- 高值异常或空值异常
- 高频维度
- 最近更新时间

这个摘要块应该是机器生成、低成本维护、稳定可复用的。

它的目标不是替代全文，而是作为“默认第一视图”。

## 迁移策略

不要一次性把所有现有库都改类型。

推荐迁移顺序：

1. 现有所有库默认保持 `document`
2. 只把明显的数据资料库手动切成 `data`
   - `order`
   - `inventory`
   - 部分 `invoice`
   - 未来专门的数据样本库
3. 观察聊天和报表输出是否更稳定
4. 再决定是否扩展更多数据型库

这样不会影响当前：

- 招标文件链
- 合同链
- 简历链
- PPT 解析链

## 对 OpenClaw 的约束边界

需要保持下面这条原则不变：

- 系统只负责供料优先级
- 不负责决定模型如何使用文档或数据

所以不应做这些事情：

- 不要在宿主层写“数据型库必须按表格回答”
- 不要在宿主层写“文档型库必须先摘要再回答”
- 不要把 `libraryType` 做成新的重编排入口

正确做法是：

- `document` -> 把全文放前面
- `data` -> 把结构摘要放前面
- 其余交给 OpenClaw

## 风险

### 风险 1：把 `libraryType` 和 `bizCategory` 混用

后果：

- 一份订单 CSV 既像 `order`，又会被误认为一定是 `data`
- 一份合同统计表又可能业务上属于 `contract`

缓解：

- 明确 `libraryType` 只决定供料策略
- 不参与业务分类本身

### 风险 2：数据型库结构摘要不稳定

后果：

- 实际还是退回全文块
- 体验没有提升

缓解：

- 先只覆盖 CSV / XLSX / JSON
- 第一版不要追求所有格式统一

### 风险 3：现有输出链受影响

后果：

- 报表生成突然换了供料风格

缓解：

- 先保持所有现有库默认 `document`
- 只在手动切换到 `data` 后才启用结构摘要优先

## 成功标准

这次升级算成功，需要满足：

1. 数据中心页面能清晰区分文档型库和数据型库
2. 数据型库普通聊天默认先看到结构摘要信号
3. 数据型库仍然允许全文供料，不被硬性截断
4. 文档型库原有招标、合同、简历、PPT 链路不回退
5. 宿主没有新增一层重编排，只新增供料优先级

## 推荐执行顺序

1. 先加 `libraryType`
2. 再改数据中心 UI 与接口
3. 然后补数据型库结构摘要块
4. 最后接到聊天和报表供料顺序

不要先改聊天主链，再补库定义。

先把库类型落稳，后面的供料切换才不会漂。
