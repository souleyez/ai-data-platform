# 知识处理与输出架构计划

日期：2026-03-26

## 1. 目标

把系统后续演进明确拆成三条独立主线，避免再出现“上传、解析、分组、问答、报表输出”互相牵连、反复翻改的问题。

三条主线分别是：

1. 文档解析与结构化存储
2. 云端对话与知识库输出编排
3. 模板化输出与持续优化

最终目标：

- 上传和扫描后的文档可以快速入库、快速可见
- 文档进库后可以异步做更深层的结构化增强
- 平时普通问答始终直接走云端模型
- 只有用户明确触发时，才进入“按知识库输出”的流程
- 输出结果遵循模板，并能在后续对话中持续改进
- 系统对 OpenClaw 保持零侵入，后续 OpenClaw 升级时我们无需修改其本体代码

## 2. 核心约束

这是本方案最重要的边界。

### 2.1 不修改 OpenClaw 本体

后续所有增强都必须放在以下位置：

- 本项目自己的 API 适配层
- 本项目自己的数据层
- 本项目自己的任务队列/后台任务
- 本项目自己的 workspace skills
- 本项目自己的前端交互层

明确禁止：

- 修改 OpenClaw 源码
- 修改 OpenClaw 内置升级流程
- 在 OpenClaw 安装目录中直接打补丁
- 把本项目关键逻辑写死进 OpenClaw 本体

这样做的目的：

- OpenClaw 升级时可直接同步升级
- 我们的系统逻辑可独立演化
- 服务器、本地、未来新环境的迁移成本最低

### 2.2 本地能力和云端能力分层

本地负责：

- 文件发现
- 文件读取
- 文档解析兜底
- 索引
- 检索
- 缓存
- 任务排队
- 失败重试

云端负责：

- 深层理解
- 复杂归纳
- 高质量结构化
- 需求总结
- 模板化生成

### 2.3 普通对话与知识库输出分离

- 普通对话不被知识库逻辑打断
- 知识库输出必须通过显式入口触发
- 输出前允许用户确认和修改需求

## 3. 三层架构

### 3.1 L1：文档解析与结构化存储

这是底层能力层，只回答一个问题：

“文档被系统学懂到什么程度？”

这层不关心首页对话，不关心输出模板。

#### 3.1.1 双阶段解析

文档解析固定拆成两段：

1. `quick parse`
2. `deep parse`

#### 3.1.2 quick parse 目标

触发时机：

- 上传文档
- 扫描新文档

要求：

- 快速返回
- 不阻塞上传
- 结果立即可见

输出内容：

- 基础文本
- 基础摘要
- 基础主题标签
- 初步 `schemaType`
- 初步 `structuredProfile`
- 初步分组建议
- `parseStage = quick`

用途：

- 文档中心立即显示
- 支持基础搜索
- 支持进入“未分组”

#### 3.1.3 deep parse 目标

触发时机：

- 文档进库后异步执行
- 未分组文档二次处理时执行
- 用户手动要求详细解析时执行

输出内容：

- 高质量摘要
- 证据块 `evidenceChunks`
- 实体 `entities`
- 结论 `claims`
- 表格片段
- 更准确的 `schemaType`
- 更完整的 `structuredProfile`
- 二次分组建议
- `parseStage = detailed`

要求：

- 不影响 quick parse 成功
- 云端失败不影响文档存在
- 只影响“增强质量”

#### 3.1.4 建议数据结构

每篇文档都固定保留两类结果：

- `quickResult`
- `detailedResult`

对外展示时：

- 优先 `detailedResult`
- 不存在时回退到 `quickResult`

建议附加状态字段：

- `parseStage`
- `parseStatus`
- `detailParseQueuedAt`
- `detailParsedAt`
- `cloudEnhancedAt`

### 3.2 L2：云端对话与知识库输出编排

这一层只回答一个问题：

“什么时候应当从普通对话切到知识库输出？”

#### 3.2.1 普通问答

- 默认全部走云端
- 不做本地知识库编排
- 不做本地追问
- 不做本地场景猜测

#### 3.2.2 知识库输出入口

只有一个显式入口：

- 按知识库输出

用户点击后，执行流程：

1. 读取最近 `3-5` 句有效上下文
2. 云端模型把上下文整理成一条可执行需求
3. 弹出确认框
4. 用户可修改需求
5. 用户确认后，才真正执行知识库输出

#### 3.2.3 知识库输出执行

执行时：

- 优先使用知识库内容
- 检索结果先由本地整理
- 高质量证据再交给云端生成最终结果

这一层不负责文档解析深度提升，只负责：

- 需求整理
- 触发知识库检索
- 组织输出请求

### 3.3 L3：模板化输出层

这一层只回答一个问题：

“结果应该按什么固定方式输出？”

建议每个知识库可绑定多个模板：

- 表格
- 静态页
- PDF
- PPT

模板建议拆成两部分：

1. 固定结构
2. 可调参数

固定结构示例：

- 标题
- 字段顺序
- 必须展示的列
- 必须引用的证据
- 固定章节结构

可调参数示例：

- 面向对象
- 语气
- 重点维度
- 排序方式
- 是否保守表达

后续对话中用户说：

- “把这个报表改成按品牌排序”
- “多加一列营销建议”
- “这个版本给客户看，语气更克制一点”

本质上修改的是模板参数，而不是临时硬改一次结果。

## 4. OpenClaw 的接入方式

### 4.1 推荐使用方式

OpenClaw 在本项目里承担三类职责：

1. 深度文档解析
2. 知识库输出需求整理
3. 模板化结果生成

### 4.2 推荐技能策略

不建议把核心主链建立在第三方技能商店技能上。

更稳的方式是：

- 核心能力使用本项目自己的 workspace skills
- 技能商店只作为参考或补充

推荐自建三个技能：

1. `document-deep-parse`
2. `knowledge-output-planner`
3. `template-output-renderer`

#### document-deep-parse

输入：

- quick parse 结果
- 原文文本
- 文档元信息

输出：

- 高质量摘要
- evidenceChunks
- entities
- claims
- schemaType
- structuredProfile

#### knowledge-output-planner

输入：

- 最近 3-5 句上下文
- 用户当前问题

输出：

- 一条可执行需求
- 输出类型
- 优先知识库建议

#### template-output-renderer

输入：

- 用户确认后的需求
- 模板定义
- 高质量知识库证据

输出：

- 表格
- 静态页
- PDF/PPT 大纲

### 4.3 对技能商店的态度

技能商店可作为参考来源，重点关注：

- Data Processing
- Docs & Sheets
- Web & Research

但核心链路不直接依赖第三方技能，原因是：

- 可控性不足
- 升级边界不清晰
- 安全审计成本更高

## 5. 向量化方案

本阶段只做方案，不直接开发。

### 5.1 为什么要做

向量化的主要价值在于：

- 提升跨表达召回能力
- 提升长 PDF 深层片段命中
- 提升同义表述、隐式相关表述的检索效果
- 为专家级表格和深层输出提供更强证据基础

### 5.2 为什么现在不立即开发

当前更大的瓶颈还在于：

- `schemaType` 打准
- `deep parse` 稳定
- 按知识库输出主链稳定

如果现在直接上向量库，容易把问题复杂化，而不是解决主要矛盾。

### 5.3 建议技术路线

#### 第一阶段：混合检索骨架

- 关键词召回
- 规则召回
- 元数据过滤
- rerank

#### 第二阶段：向量索引

只对 `detailed` 结果建立向量，不对原始全文整篇建向量。

推荐向量粒度：

- evidenceChunk
- 表格片段
- 结构化摘要片段

#### 第三阶段：混合召回

- keyword recall
- vector recall
- rerank

#### 第四阶段：云端消费高质量证据

最终只把最强证据交给云端，而不是整篇全文。

### 5.4 向量化的数据边界

向量索引存储内容建议仅包括：

- 文档 id
- 分组 id
- schemaType
- chunk 文本
- chunk 摘要
- chunk 标签
- embedding

不要把整份原文粗暴塞入向量层。

## 6. 推荐实施顺序

### M1：稳定 quick parse

目标：

- 上传快反馈稳定
- 未分组逻辑稳定
- quick parse 结构固定

交付：

- `quickResult`
- 初步分组建议
- 立即入库可见

### M2：建立 deep parse 任务流

目标：

- quick parse 与 deep parse 解耦
- 入库后异步深度解析

交付：

- 详细解析任务队列
- 失败重试
- 解析阶段状态

### M3：接入 OpenClaw workspace skills

目标：

- 用 skill 做深度结构化
- 不修改 OpenClaw 本体

交付：

- `document-deep-parse`
- `knowledge-output-planner`

### M4：稳定知识库输出链

目标：

- 普通问答与知识库输出完全分离
- 知识库输出有确认环节

交付：

- 按知识库输出按钮
- 需求整理
- 用户确认
- 执行输出

### M5：模板系统上线

目标：

- 输出稳定
- 可持续调优

交付：

- 模板定义结构
- 模板参数结构
- 模板化表格与静态页

### M6：向量化接入

目标：

- 提升召回质量
- 提升深层文档利用率

交付：

- 向量索引
- 混合检索
- rerank 合流

## 7. 近期优先事项

按当前项目状态，建议优先级如下：

1. 固定 quick / deep parse 双阶段数据结构
2. 用 OpenClaw workspace skill 做 deep parse
3. 稳定“按知识库输出”的确认流程
4. 引入模板系统
5. 最后接向量化

## 8. 验收标准

### 文档解析

- 任意文档上传后 10 秒内可见
- 云端失败不影响文档入库
- `deep parse` 完成后详情和分组明显提升

### 对话编排

- 普通对话不被知识库逻辑打断
- 知识库输出只通过按钮触发
- 用户确认后才执行

### 模板输出

- 同类需求输出结构稳定
- 用户可通过对话持续优化模板
- 结果能沉淀为系统报表资产

### 升级与维护

- OpenClaw 升级时，无需修改其本体
- 本项目增强逻辑全部保留在自身代码仓库中

## 9. 结论

最稳妥的路线不是继续混合式修补，而是：

- 先固定三层边界
- 再按里程碑逐步推进
- 对 OpenClaw 保持零侵入
- 核心能力通过本项目自己的 workspace skills、任务流和模板系统完成

这样系统才能持续进步，而不是每增加一个能力，就重新打乱上传、解析、问答和输出链路。
