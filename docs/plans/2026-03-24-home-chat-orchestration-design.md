# 首页对话编排设计

日期：2026-03-24

## 目标

把首页对话区从“前端猜场景 + 后端拼回答”的模式，改成“云端模型主导编排，本地知识库负责供给材料”的模式。

设计目标：

- 首页所有对话默认优先走 OpenClaw 云端模型
- 除系统修改类请求外，其余内容都允许自然对话
- 是否调用知识库、调用哪个知识库、输出什么形态，由云端模型判断
- 本地文档解析结果只作为原始材料层，不直接决定最终回答或报表结构
- 报表、表格、静态页、PPT/PDF 等输出统一由 `/api/chat` 协议驱动

非目标：

- 本阶段不重做文档解析器
- 本阶段不引入新的桌面能力
- 本阶段不新增复杂权限系统

## 当前问题

当前首页对话链路存在几个结构性问题：

- 前端通过关键词推测场景，容易误判
- 后端 `orchestrator.ts` 同时承担意图判断、知识库范围控制、云端调用、fallback、报表逻辑，职责过重
- 本地解析结果直接影响最终输出，缺少云端二次归纳层
- 前端右侧报表区和 `/api/chat` 的协议没有真正统一，导致首页容易堆前端特殊逻辑

更准确地说，系统当前有“知识库”和“模型”，但缺少一个明确的“编排协议”。这会让首页主工作台继续扩展时越来越难维护。

## 设计原则

采用三层结构：

1. 本地知识层  
负责文档扫描、解析、切片、摘要、实体、表格片段、证据提取。

2. 云端编排层  
负责判断用户意图、是否需要知识库、需要哪些知识库、输出什么类型。

3. 结果输出层  
负责把最终结果统一包装成普通回答、表格报表、静态页、PPT/PDF 等前端可消费对象。

关键原则：

- 云端模型是默认入口，不是补充入口
- 本地规则只做安全边界和知识检索，不做最终业务决策
- 任何涉及系统修改、写操作、外部执行的请求都必须单独拦截
- 前端不再猜业务场景，只消费统一响应协议

## 第一阶段范围

第一阶段只做协议和编排骨架，不做所有高级产物：

- 首页对话全部默认走 OpenClaw
- `/api/chat` 升级为统一协议
- 云端模型先做意图判断
- 命中知识库时再走本地检索
- 云端基于文档上下文输出最终回答或表格
- 前端右侧报表列表只认统一的 `output` 结构

第一阶段不做：

- 真正的静态页/PPT/PDF 文件生成
- 多轮复杂编辑工作流
- 细粒度用户权限
- 长会话历史记忆策略

## 新的 `/api/chat` 协议

请求：

```json
{
  "prompt": "帮我看一下合同库里付款条款风险，并出一份表格",
  "sessionUser": "optional-user-id",
  "conversationId": "optional-thread-id"
}
```

响应：

```json
{
  "mode": "openclaw",
  "intent": "report",
  "needsKnowledge": true,
  "libraries": [
    { "key": "contract", "label": "合同协议" }
  ],
  "answer": {
    "title": "合同付款条款风险",
    "content": "先给结论，再给简短说明。"
  },
  "output": {
    "type": "table",
    "title": "合同付款条款风险清单",
    "content": "表格说明文本",
    "table": {
      "columns": ["文档", "风险点", "风险等级", "建议"],
      "rows": [
        ["采购合同A", "账期未封顶", "中", "补充最大账期约束"]
      ]
    }
  },
  "references": [
    {
      "id": "doc_xxx",
      "name": "采购合同A.pdf",
      "summary": "合同付款周期为净45天，未约定最长期限。"
    }
  ],
  "guard": {
    "requiresConfirmation": false,
    "reason": ""
  },
  "orchestration": {
    "selectedBy": "cloud",
    "fallbackUsed": false
  }
}
```

约束：

- `mode`：`openclaw | fallback`
- `intent`：`general | knowledge_qa | report | file_output | system_change`
- `output.type`：`answer | table | page | ppt | pdf`
- 如果没有结构化输出，`output.type = answer`
- 如果属于系统修改请求，必须返回 `guard.requiresConfirmation = true`

## 编排流程

### 1. 云端意图分类

先给 OpenClaw 一个轻量编排 prompt，只要求返回 JSON 决策：

- 当前意图是什么
- 是否需要知识库
- 优先选哪些知识库
- 输出类型是什么
- 是否需要补充澄清
- 是否属于系统修改类请求

这一阶段不让云端直接写最终长回答，只做决策。

### 2. 本地知识检索

当 `needsKnowledge = true` 时：

- 根据云端给出的 `libraries`
- 在对应知识库分组中检索文档
- 抽取：
  - `summary`
  - `excerpt`
  - `topicTags`
  - `claims`
  - `intentSlots`
  - `table fragments`

本地层输出统一的 `knowledgeContext`，但不直接产最终答案。

### 3. 云端二次归纳

把 `prompt + intent decision + knowledgeContext` 再发给 OpenClaw，要求它：

- 先总结结论
- 再按目标输出类型组织结果
- 如果是表格，就返回明确列头和行内容
- 如果是普通问答，就返回简洁答案和引用依据
- 如果证据不足，就明确指出不足

### 4. 输出封装

后端把云端结果规范化为统一协议：

- `answer`
- `output`
- `references`
- `guard`

前端首页只认这个结构，不再做场景猜测。

## 系统修改请求的边界

需要单独拦截的请求包括：

- 删除、覆盖、移动本地文件
- 改配置、改模型、改系统参数
- 执行命令
- 对外部系统做写入

这些请求允许对话，但不允许直接执行。统一返回：

```json
{
  "intent": "system_change",
  "guard": {
    "requiresConfirmation": true,
    "reason": "该请求涉及系统修改，需显式确认后再执行。"
  }
}
```

首页表现：

- 对话里给出说明
- 不自动触发任何动作
- 后续如要实现确认流，再单独加确认入口

## 前端改造点

首页对话区只做三件事：

1. 发 `prompt`
2. 渲染 `answer`
3. 如果 `output.type !== answer`，沉淀到右侧报表列表

右侧报表列表：

- `table`：显示表格正文
- `page`：显示链接操作
- `ppt/pdf`：显示下载操作
- 默认只显示标题，点开才展开详情

前端不再：

- 靠关键词切换场景
- 本地推断报表类型
- 手写右侧面板业务映射

## 后端改造点

建议把当前编排拆成 4 个模块：

- `chat-intent-router.ts`
  - 专门调用 OpenClaw 做意图判断
- `knowledge-context-builder.ts`
  - 专门从知识库生成云端可消费上下文
- `chat-output-normalizer.ts`
  - 把云端返回标准化成前端协议
- `chat-guard.ts`
  - 专门判断系统修改类和安全边界

`orchestrator.ts` 逐步缩成调度层，而不是继续堆逻辑。

## 回退策略

如果 OpenClaw 不可用：

- `mode = fallback`
- 普通问答返回简短兜底说明
- 知识库问答可继续使用本地 evidence-based fallback
- 报表输出降级为普通文本，不强行生成结构化报表

这样能保证系统稳定，但不会再把 fallback 当主路径。

## 第一阶段实施顺序

1. 先定义 `/api/chat` 新协议
2. 实现云端意图分类调用
3. 接入知识库检索上下文构建
4. 实现云端二次归纳输出
5. 首页对话区改成只消费新协议
6. 右侧报表列表按 `output.type` 接入
7. 保留 fallback，但改为兜底路径

## 验证标准

第一阶段完成后，至少要验证这几类对话：

- 普通闲聊或业务讨论，不调用知识库
- 指定知识库问答，例如“看合同库里付款条款风险”
- 要求生成表格报表
- 要求生成静态页/PPT/PDF 的占位输出
- 明确提出系统修改请求时被拦截
- OpenClaw 不可用时系统仍能返回可理解结果

## 建议结论

先做第一阶段，不要一口气做所有产物生成。

最关键的不是先做 PPT 或 PDF，而是先把“首页对话 -> 云端判断 -> 本地知识增强 -> 统一输出协议”这条主链打通。只要这条主链稳定，后面加静态页、PPT、PDF 都只是 `output.type` 的扩展，不会再把首页重新做乱。
