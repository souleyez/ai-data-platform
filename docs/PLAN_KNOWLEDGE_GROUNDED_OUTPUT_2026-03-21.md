# 文档中心资料优先输出方案

日期：2026-03-21

## 目标

将当前系统从“文档级摘要匹配 + 模型泛化回答”升级为“证据块级检索 + 引用驱动回答 + 模板可回溯输出”。

目标效果：

- 回答优先基于文档中心资料，而不是弱关联泛化输出
- 每个关键结论都能指出来自哪份文档、哪段证据
- 奶粉配方建议、报表输出、数据可视化等结果都能回溯到知识库资料
- 当知识库证据不足时，系统明确说明不足，而不是补全超出文档范围的内容

## 当前状态

当前系统已经具备这些基础能力：

- 文档解析后可生成 `summary`、`excerpt`、`topicTags`、`groups`、`contractFields`
- 对话前会按知识库分组做范围约束
- 命中文档后，会把摘要和摘录作为上下文传给本地 AI 或云端模型
- 奶粉配方类问题已有固定模板输出能力

当前主要问题：

- 检索粒度仍然是“文档级”，不是“证据块级”
- 命中主要依赖文件名、摘要、摘录和关键词打分
- 送给模型的是压缩后的摘要信息，不是精确命中的段落证据
- 模板输出更多是本地规则驱动，文档更多作为辅助背景而非主驱动
- 回答和报表目前缺少稳定的“结论 -> 文档 -> 证据块”回链

## 核心判断

当前链路已经有“解析 -> 存储 -> 粗召回 -> 上下文注入”的基础，但还没有达到“资料优先输出”的目标状态。

因此，用户会感知到：

- 回答偶尔能命中文档，但引用关联偏弱
- 输出内容看起来有知识库参与，但不像是严格从文档中长出来的
- 模板结果和具体文档之间的绑定仍不够强

## 推荐方案

### 1. 文档改为 chunk 级结构化存储

每份文档解析完成后，除了保留当前整篇级字段，还需要新增 chunk 存储。

每个 chunk 建议保存：

- `documentId`
- `chunkId`
- `text`
- `sectionTitle`
- `order`
- `charLength`
- `topicTags`
- `groups`
- `parseMethod`

建议分段方式：

- 先按标题、空行、列表、自然段做语义切分
- 超长段再按长度切片
- 单个 chunk 控制在适合回答引用的长度范围

这样可以避免整份文档只有一个摘要，导致召回过粗。

### 2. 检索从文档级改为 chunk 级

用户提问后，检索链路改成：

1. 先按知识库分组过滤候选文档
2. 再按 chunk 做相关度打分
3. 召回前 5-8 个高相关证据块
4. 将这些证据块而不是整篇摘要送入回答链

第一阶段不一定要上向量库，可以先做：

- 关键词打分
- 标题命中
- 标签命中
- chunk 内容命中
- 分组加权

这样改动较小，但效果会比现在明显更强。

### 3. 回答改为证据驱动

对话输出应改成：

- 先给结论
- 再列证据
- 再说明不足或不确定点

每条证据至少要带：

- 文档名
- 证据摘录
- 可选的段落标题或 chunk 序号

如果没有足够证据，不允许模型自由补全，应明确回答：

- 当前知识库未覆盖
- 当前资料不足以判断

### 4. 模板输出改为“证据填充模板”

奶粉配方模板、报表模板、静态页模板，都要从“规则直接出结果”升级为“证据抽取后填模板”。

建议做法：

- 先抽取支持当前问题的证据块
- 再将证据映射到模板字段
- 对没有证据支撑的字段标记为“待补充”或“不足以判断”

这样才能让模板输出真正由知识库驱动，而不是只有一个模板外壳。

## 分阶段实施

### 第一阶段：证据链 MVP

目标：

- 让回答明显更像“基于资料”

工作项：

- 新增 chunk 存储结构
- 文档解析后自动生成 chunk
- 提问时改用 chunk 检索
- 回答里显示来源文档和证据摘录

验收：

- 同一个问题能稳定引用 1-3 份相关文档
- 回答和引用的内容高度相关
- “资料不足”场景能明确拒绝泛化输出

### 第二阶段：模板证据化

目标：

- 让配方建议和报表输出真正由知识库证据驱动

工作项：

- 奶粉配方模板接入 chunk 证据
- 报表输出记录所使用的文档和证据块
- 历史输出增加来源追踪信息

验收：

- 模板里的关键项都能回溯到文档证据
- 报表中心可显示来源知识库和来源资料

### 第三阶段：更强检索与结构化

目标：

- 进一步提升复杂问题命中和输出稳定性

候选工作项：

- 向量检索
- 实体抽取
- 关系抽取
- 学术论文结构化索引

说明：

这一阶段不是当前 MVP 必需项，建议在前两阶段稳定后再推进。

## 技术改造点

建议优先改这些文件或模块：

- `apps/api/src/lib/document-parser.ts`
  - 在解析结果后补 chunk 生成
- `apps/api/src/lib/document-store.ts`
  - 增加 chunk 级存储与 chunk 检索
- `apps/api/src/lib/orchestrator.ts`
  - 将当前回答上下文从整篇摘要改为证据块列表
- `apps/web/app/components/ChatPanel.js`
  - 展示回答引用来源
- `apps/web/app/reports/*`
  - 报表结果中展示来源资料

## 服务器资源影响

第一阶段资源影响较低：

- 存储会增加，因为需要保存 chunk
- 检索耗时会略有增加
- 不需要额外付费 API

第二阶段资源影响中等：

- 模板证据映射会增加一些处理逻辑
- 报表结果会保存更多来源元数据

第三阶段资源影响中高：

- 如果上向量库或更深结构化，会增加额外索引成本

## 验收标准

达到以下标准，可认为“文档资料优先输出”基本成型：

- 普通问答能稳定基于文档中心资料输出
- 回答能展示清晰引用来源
- 模板结果能回溯到具体资料
- 非知识库覆盖内容不会被强行生成
- 文档中心资料与聊天输出之间的关联明显增强

## 建议的实际推进顺序

推荐顺序：

1. 先做 chunk 存储
2. 再做 chunk 检索
3. 再做引用展示
4. 最后再做模板证据化

不建议一开始就直接上向量库或知识图谱，因为当前最短板的问题并不是“没有高级检索”，而是“还没建立证据块级链路”。

## 结论

当前系统已经具备“资料参与回答”的基础，但还没有达到“资料主导回答”的状态。

下一步最值得做的是：

- 文档 chunk 化
- chunk 检索
- 回答引用回显

这是最小但收益最高的一轮升级，也最符合当前产品阶段。
