# PDF OCR 与网页正文采集计划

## 目标

- 提升 PDF 文档解析成功率，尤其是学术论文型 PDF 和扫描型 PDF。
- 提升网页采集质量，只保留正文内容，减少导航、广告、页脚和模板噪音。
- 提升知识库入库内容质量，帮助后续对话命中、模板输出和报表生成更加稳定。
- 保持系统可独立运行，不影响当前 120 服务器演示环境的稳定性。

## 技术选型

### 文档解析

- PDF 文本提取主链：`pypdf`
- PDF 兼容链：现有 `pdf-parse` 修正入口
- PDF OCR fallback：`Tesseract OCR`
- Office 文档：沿用现有 `mammoth` 和 `xlsx`

### 学术论文增强

- 候选方案：`GROBID`
- 当前建议：暂不接入，等第一阶段稳定后再评估

### 网页正文采集

- 主提取器：`Mozilla Readability`
- fallback：`Trafilatura`
- 原始 HTML：保留为审计附件，不作为主要知识文本

### 质量控制

- 为文档和网页增加解析质量判定
- 重点指标：
  - 正文长度
  - 空白率
  - 可读段落数
  - 链接占比
  - 是否疑似扫描件
  - 是否疑似列表页/导航页

## 分阶段里程碑

### 第一阶段：基础增强

- PDF 改成“文本提取优先，失败再 fallback”
- 网页采集增加“只提正文”
- 文档和网页增加解析状态标记
- 不改现有业务交互，只提升底层解析质量

目标：
- 大多数正常论文 PDF 可提取正文
- 大多数文章型网页可只入正文

### 第二阶段：OCR 补强

- 仅对失败或低质量 PDF 启用 OCR
- 支持后台重跑失败文档
- 页面展示“OCR 解析”来源

目标：
- 扫描型 PDF 能进入 OCR 链路
- 失败文件可后台补救，不依赖手工重传

### 第三阶段：论文结构化增强

- 评估并接入 `GROBID`
- 提取标题、摘要、章节和参考文献
- 为知识库问答和报表生成提供更强结构化输入

目标：
- 论文型问答更容易命中摘要、结论和章节信息
- 学术资料更适合作为固定模板输入

## 服务器资源影响

### 第一阶段

- 资源影响：低
- 主要表现：
  - CPU 小幅增加
  - 内存影响可控
  - 对当前 120 服务器风险较低

### 第二阶段

- 资源影响：中
- 主要表现：
  - OCR 会显著增加 CPU 占用
  - 不适合全量默认开启
  - 需要限制在失败或低质量文档上触发

### 第三阶段

- 资源影响：中高
- 主要表现：
  - 如果接入 `GROBID`，会增加独立服务和更多内存占用
  - 更适合论文量持续增长后再部署

## 验收标准

- 正常论文 PDF 大多数可成功提取正文
- 扫描型 PDF 至少可进入 OCR 重试链
- 网页采集结果以正文为主，模板噪音明显下降
- 对话引用证据更贴近正文，而不是页面杂项
- 不影响当前 120 服务器的演示稳定性

## 当前建议

- 今天先不继续改 120 服务器核心链路，优先保证演示稳定。
- 下一轮优先推进：
  1. `pypdf + 条件 OCR`
  2. `Readability` 正文提取
  3. `Trafilatura` fallback
- `GROBID` 放到后续论文增强阶段评估，不作为当前第一优先级。
